Your host:  Alex Bridge

What is OWASP?

  • An online community dedicated to web application security
  • Provides free guidance and tools for improving security

Community is made up of people & organisations around the world with an interest in security

Some of these orgs are commercial, but OWASP's advice is independent & vendor-neutral

Guidance referenced in legislation internationally, including UK's

What is the OWASP Top Ten?

  • A list of the most critical web application security risks
  • Guidance on how to avoid each of them in your web applications

NB it's not an exhaustive list - but paying attention to the top ten will take you a long way towards securing your applications

1. Injection

Made possible when user input is passed directly to back-end systems (e.g. databases)

Can lead to data leaks, corruption and system takeover

Commonplace & easy to exploit

Recent example: Drupalgeddon, a flaw in Drupal's DB library.
This left all Drupal 7 sites vulnerable to complete takeover by hackers, and there was a huge scramble to patch sites.

Injection prevention

  • Use parameterised APIs if available
  • If no safe API, ensure all inputs are escaped
  • Use whitelists of acceptable inputs over free text wherever possible

SQL injection most widely-known flaw

But can happen elsewhere, e.g. SMTP headers, LDAP

3. Cross-Site Scripting (XSS)

Made possible when malicious input is output directly in pages to users

Attacker can execute scripts in a victim’s browser, making your site a source of malware

This is the most widespread security flaw in web applications

Executing scripts means they can hijack user sessions & browsers, giving them the ability to do anything the user can do, and potentially tricking them into revealing sensitive information

XSS prevention

  • Use sanitisation routines if available
  • Escape all data from users which might be output to a page
  • Use whitelists of acceptable inputs over free text wherever possible
Might look familiar to injection recommendations - DON'T TRUST USER INPUT!

5. Security misconfiguration

This covers a wide variety of problems, including

  • Security settings in installed software not set correctly
  • Unnecessary software running
  • Insecure or unchanged-default passwords/accounts

Security settings - e.g. leaving Apache directory listings enabled

Hacking Team - a company that sells intrusion and surveillance capabilities to governments, law enforcement agencies and corporations You might expect them to have their own house in order...

Avoiding security misconfiguration

There's no one silver bullet - you need to build security into day-to-day work practices

  • Have a repeatable and automated hardening process for setting up environments
  • Scan and audit systems regularly
  • Enforce separation between system components to minimise impact of compromises
Repeatable hardening process - this is where provisioning technologies like Ansible, Chef, Puppet etc. come in

6. Sensitive Data Exposure

Sensitive data might include passwords, credit card numbers, other personal information.

This might be exposed by SQL injection, server exploit or a man-in-the-middle attack

MITM could occur if pages handling sensitive data are sent in clear text, rather than encrypted using e.g. HTTPS

Avoiding sensitive data exposure

  • Don't store what you don't need
  • Ensure all sensitive data is securely encrypted, both at rest and when transmitted
  • Disable autocomplete for sensitive forms & page caching for sensitive pages

Audit of Ashley Madison leak showed that passwords & credit cards were secured ok, but names, addresses & sexual preferences weren't.

This has opened people up to embarrassment, blackmail, and potentially persecution.

9. Using Components with Known Vulnerabilities

This could potentially undermine all other security measures

Known vulnerabilities can be identified by automated scanning tools

Scanning tools means you can be hacked without a concerted effort on the part of the hacker

Avoiding use of vulnerable components

  • Record all software you're using, including version numbers
  • Monitor these for updates
  • Have a process to quickly roll out updates
  • Have a policy for what software components are/aren't acceptable

Monitoring might include subscribing to mailing lists or code repositories

Process for rolling out updates might include configuring servers to auto-apply security patches; Ansible etc can also help again here

Policy for software might enforce e.g. no alpha/beta software; code having passed audit

Accessibility in web design

matth@cogapp.com

What is accessibility?

Removing barriers that prevent access to website content and functionality

Inclusivity and diversity

  • Same idea as elsewhere
  • Inclusivity and diversity - this sector is very bad at this, we have to get better and foregrounding accessibility helps to raise awareness of this

How should we think about accessibility?

Users are not simply disabled or non-disabled

There are huge range of disabilities: cognitive, motor, sensory

Disability may be temporary

If we make sure websites are accessible everyone benefits

Legal duties

  • Can't make any assumptions about users
  • Temporary: eg. broken wrist ... or blind and using screen reader
  • A11y testing shouldn't be tacked on to the end of a project but embedded within processes
  • US receipt of public money = Section 508 compliance
  • We need to be careful - National Federation of the Blind v. Target Corp class action $6mill

How do we know that a website is accessible?

WCAG (Web Content Accessibility Guidelines)

4 main principles: perceiveable, operable, understandable and robust

12 guidelines

65 testable suceess criteria

A, AA, AAA

Section 508

  • As a developer not using assistive technology how do we test for accessibility
  • 2008 W3C WCAG published by Web Accessibility Initiative (WAI)
  • There are 12 guidelines grouped under four principles (perceiveable, operable, understandable and robust)
  • there are 65 testable suceess criteria covering the guidelines.
  • A - AAA - level of compliance
  • AAA very hard - concerning eg. transcripts of live video and audio

Some WCAG success criteria

Provide text alternatives for non-text content - A

Links must have descriptive text content - A

A sign language interpreter is included for recorded media - AAA

Text to background contrast ratio must be at least 4.5:1 - AA

  • alt text read by screen readers - text representation of visual image
  • Not just developers - designers as well! and producers

How do we test accessibility

Chrome Accessibility Audit

WAVE browser plugin

HTML Codesniffer

  • hard to go through all WCAG guidelines and criteria before delivering every feature
  • some tools we have been using
  • Chrome - not as comprehensive as other tools but nicely highlights elements
  • WAVE - tests contrast ratio, header hierarchy
  • Codesniffer - nice interface and most comprehensive

Automated testing

grunt-accessibility

  • tenon.io comprehensive api which can be hooked into build process
  • can't replace manual testing completely eg. alt text exists but does it
